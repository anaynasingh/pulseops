**Goal:** Make Claude mode work on the live Railway deployment — deploy `claude-bridge/` as its own Railway service (subscription `CLAUDE_CODE_OAUTH_TOKEN`, no Anthropic API key) with both the pulseops and m365 MCP servers attached, so the user can say "read my meeting transcripts / emails and create tasks" and have it act on the real board.
**CHARTER alignment:** Objectives "Email Intelligence working" and "Meeting Intelligence working" (Claude path), plus Hard Spec "MCP servers for external integrations (M365, pulseops-native)"; includes the Orchestrator-approved Hard Specs amendment to the "OpenRouter (GPT-4o) for all LLM inference" line to sanction Claude-via-bridge as a second inference path.
**Streams:** Three write-disjoint streams, built sequentially A → B → C — A (bridge + m365 code), B (Dockerfile + Railway config), C (docs, runbook, charter amendment). Write-sets verified pairwise disjoint; B and C consume contracts fixed in this plan, not files written by A. Build order is A → B → C sequential (all orchestrator-direct, single session): write-sets are disjoint but the streams are semantically dependent.
