- High: `confirm_intake` auth/access is under-specified. ai.py:171 uses `get_current_user`/`get_db`, and the plan moves direct Project/Task creation plus `target_project_id` attachment into that endpoint. The route enumeration only says missing target gives 404; it does not require writer eligibility or permission to add tasks to the chosen project. Literal implementation risks authenticated users creating work or injecting tasks into arbitrary project UUIDs.

- High: The regression gate has a concrete port mismatch and a determinism gap. test_regression.py:4 says 8001, but line 13 points at `localhost:8002`; the integration gate says run against 8001. Also, the seven intake cases need deterministic subtasks/type/status, but the HTTP intake path calls live AI via ai.py:126. The plan does not define how tests create stable pending intakes without external AI variance.

- High: `ConfirmIntakeResult.tasks: [TaskOut]` has a serialization trap. `TaskOut` includes relationship fields (`assignee`, `project`) in schemas.py:213, while current confirm only reloads the project graph before returning in ai.py:214. The plan says to return created `TaskOut`s but does not require reloading created tasks with the same relationship discipline. With async SQLAlchemy, lazy relationship access during Pydantic serialization can fail or produce inconsistent response shape.

- Medium: The "parallel streams, no sequencing dependency" claim hides a runtime trap. The current confirm client is untyped in api.ts:90, and the current backend request model has no new routing fields in schemas.py:345. If frontend lands first, task override fields can be accepted by the call path but ignored by the old backend behavior, yielding a 201 project creation. Typecheck/build will not catch this because `confirmIntake` returns `any`.

- Medium: AI classification validation is weaker than the contract. `_schema_hint` only emits field types, not enum literals, in ai_service.py:31, and `structured_completion` validates before handler fallback code can run in ai_service.py:80. If `suggested_item_type` is added as a plain required `str` in ai.py:43, missing or invalid AI output can either fail parsing or persist an invalid route hint. The acceptance criteria do not cover missing/invalid AI item type or legacy rows with null classification.

- Medium: Frontend routing state reset is not specified. The page currently resets only result/priority around analysis and discard in page.tsx:22 and page.tsx:221. Adding item type, target project, and new-project title without explicit reset criteria risks stale selection carrying across a second analysis. The confirm button is currently gated only by priority in page.tsx:214, and the plan lacks acceptance for "task existing requires a selected project" versus "explicit New project".

- Medium: Task→new parent project semantics are inconsistent with the existing override fields. Current project creation in confirm uses title, description, owner, team, tags, due date, and next action in ai.py:191. The plan's task→new route says "minimal parent Project" with title/status/priority only, while `IntakeConfirmRequest` still carries owner/team/title/description in schemas.py:345. Acceptance does not define which overrides survive for task-created parent projects.
